Welcome to this small notebook on the web.

Most developer portfolios are galleries of finished work: polished screenshots, bullet‑pointed achievements, and a carefully curated list of technologies. This site is closer to a **lab notebook**—a place to keep experiments, half‑finished ideas, and the longer stories behind projects.

## Why another blog?

There are already more programming blogs than any one person could reasonably read. The point of this one is not to be comprehensive or authoritative; it is to be specific.

I am a third‑year computer science student who spends a lot of spare time coding for fun. Some weeks that means writing small utilities or “vibe code” that never leaves my machine. Other weeks it means building:

- Tiny languages with hand‑rolled parsers.
- Redis‑style key–value stores in C, driven by curiosity about how persistence actually works.
- Minimal tools that make it easier to read code slowly instead of flailing at it quickly.

This notebook is where those explorations are documented.

## What you will (and will not) find here

You will not find exhaustive tutorials or lists of “10 tips for productivity”. There are already excellent resources for that. Instead, the posts here try to answer narrower questions, such as:

1. What does it feel like to design a programming language when you already know the implementation details?
2. How do small choices in an editor’s typography change the way you debug a complex system?
3. What trade‑offs appear when you implement a Redis clone with only a small subset of commands?

Along the way, there will be code samples, checklists, and diagrams, but the focus is always on the underlying reasoning rather than the finished artifact.

> A quiet tool does not try to look smart. It tries to make it easier for *you* to be smart, mostly by staying out of the way.

## A tiny example: thinking in commits

One recurring theme in these notes is that **good tools reinforce good habits**. Version control is a good example. A chaotic history full of `wip` and `fix` commits makes it harder to reconstruct what happened when.

Consider the difference between these two commit logs:

```text
wip
fix
try again
another attempt
```

and:

```text
feat: add quiet layout shell

- Introduce centered reading column (~70ch).
- Add typography scale for prose vs. UI.
- Define CSS tokens for ink, surface, and accent layers.
```

The second version takes slightly longer to write, but it pays you back the next time you debug an incident or try to remember *why* a change was made.<sup id="fnref:1"><a href="#fn:1">1</a></sup>

## How to read this site

If you only have a few minutes, the best place to start is usually the most recent post on the home page. Many of the essays are self‑contained, but some of the project logs build on each other. Roughly speaking:

- The **blog** contains field notes, experiments, and reflections.
- The **projects** section contains longer build logs and post‑mortems.
- The **typography sandbox** is a living test page for the design system.

## Looking ahead

In the near term, I plan to write about:

- Building a small interpreted language with a focus on error messages.
- Understanding durability and crash semantics by hand‑rolling append‑only logs.
- Designing a “quiet” terminal UI for long‑running background jobs.
