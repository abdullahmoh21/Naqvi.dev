This project started as a weekend experiment: implement a tiny expression language with a hand‑written parser, a minimal type checker, and a bytecode interpreter. The goal was not performance or features; it was to understand how real compilers behave when things go wrong.

## The shape of the language

The language—nicknamed **Glimpse**—supports:

- Integer and boolean literals.
- Let‑bindings.
- Simple arithmetic and comparison operators.
- A minimal `if` expression.

In pseudocode, a small sample program looks like this:

```text
let threshold = 10
let value = 7

if value &lt; threshold {
  value + 1
} else {
  threshold
}
```

The implementation is intentionally straightforward: a recursive‑descent parser produces an AST, which is then fed into a type checker and finally into a bytecode generator.

## The first serious bug

The interesting part of the project arrived when the compiler started accepting programs it should have rejected.

For example, this program compiled and ran:

```text
let value = 7

if value {
  1
} else {
  0
}
```

The language does not allow using integers as conditions; only booleans are legal. This meant that somewhere between the parser, type checker, and runtime, a constraint had been dropped.

> In a small codebase, inconsistent invariants are often harder to spot precisely because every individual function seems obvious in isolation.

## Tracing the invariant

The bug eventually turned out to be a missing check in the type‑checker’s `visitIf` function. A simplified version looked like this:

```ts
function visitIf(node: IfExpression): Type {
  const conditionType = visit(node.condition);
  const thenType = visit(node.thenBranch);
  const elseType = visit(node.elseBranch);

  if (!isAssignable(thenType, elseType)) {
    error("branches of if must have compatible types", node);
  }

  return thenType;
}
```

The fix was not just to add a condition:

```ts
if (!isBoolean(conditionType)) {
  error("if condition must be a boolean", node.condition);
}
```

but also to make that invariant explicit in the documentation and tests. Otherwise it would be easy to accidentally relax it again during a refactor.

## Why this belongs in a portfolio

From the outside, this project is modest: a toy compiler with a limited feature set. What makes it interesting is the **story** of how it evolved and how bugs were found and fixed.

Writing up those stories has two benefits:

1. It forces me to articulate the mental model behind the code.
2. It gives future‑me a map of what went wrong and how it was repaired.

The notes in this essay are a snapshot of that process. The implementation will almost certainly change; the invariants and the debugging lessons are the parts worth keeping.
